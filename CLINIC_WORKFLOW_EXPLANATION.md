## Overview
Your clinic has two main service workflows: **Lab Tests** and **Medications**. Both follow a similar pattern but have different processing stages.

---

## 📋 LAB TEST WORKFLOW

### Step 1: Doctor Orders Lab Test
**Location**: Doctor Dashboard → Lab Request Form

```
Doctor selects patient → Chooses lab tests → Submits order
```

**What Happens in Backend**:
```javascript
POST /api/lab-orders
- Creates LabOrder document
- Status: "Pending Payment"
- PaymentStatus: "pending"
- Creates Medical Invoice (linked to patient)
- Sends notification to Reception
```

**Database Records Created**:
1. **LabOrder**:
   ```javascript
   {
     patientId: "P-12345",
     testName: "Glucose, Fasting",
     status: "Pending Payment",  // ← Waiting for payment
     paymentStatus: "pending",
     totalPrice: 200,
     orderingDoctorId: doctorId,
     orderDateTime: new Date()
   }
   ```

2. **MedicalInvoice**:
   ```javascript
   {
     patient: patientId,
     items: [{
       itemType: "lab",
       serviceName: "Glucose, Fasting",
       unitPrice: 200,
       quantity: 1
     }],
     status: "pending",
     balance: 200  // Amount owed
   }
   ```

3. **Notification** (to Reception):
   ```javascript
   {
     type: "payment_required",
     message: "Lab test payment required for John Doe",
     recipient: "reception",
     data: {
       patientId: "P-12345",
       testName: "Glucose, Fasting",
       amount: 200
     }
   }
   ```

---

### Step 2: Patient Pays at Reception
**Location**: Reception Dashboard → Payment Queue

```
Reception sees notification → Processes payment → Patient moves to lab queue
```

**What Happens in Backend**:
```javascript
POST /api/billing/lab/payment
- Updates LabOrder status: "Pending Payment" → "Ordered"
- Updates LabOrder paymentStatus: "pending" → "paid"
- Adds payment record to invoice
- Creates invoice if doesn't exist
- Marks notification as resolved
- Patient status: "Waiting" (in queue for lab)
```

**Updated Records**:
1. **LabOrder**:
   ```javascript
   {
     status: "Ordered",        // ← Now ready for lab processing
     paymentStatus: "paid",    // ← Payment received
     paidAt: new Date(),
     invoiceId: invoiceId      // ← Linked to invoice
   }
   ```

2. **MedicalInvoice**:
   ```javascript
   {
     status: "paid",
     balance: 0,              // ← Fully paid
     amountPaid: 200,
     payments: [{
       amount: 200,
       method: "cash",
       date: new Date()
     }]
   }
   ```

---

### Step 3: Lab Technician Processes Test
**Location**: Lab Dashboard → Enter Results

```
Lab tech sees order → Collects specimen → Performs test → Enters results
```

**What Happens in Backend**:
```javascript
PUT /api/lab-tests/:id/results
or
POST /api/lab-results/submit-results
- Updates LabOrder status: "Ordered" → "Results Available"
- Stores test results
- ✅ DEDUCTS INVENTORY (Glucose Test Strips: -1)
- Creates InventoryTransaction
- Sends notification to ordering doctor
```

**Updated Records**:
1. **LabOrder**:
   ```javascript
   {
     status: "Results Available",
     results: "Glucose: 95 mg/dL (Normal)",
     resultDateTime: new Date(),
     inventoryDeducted: true,    // ← Flag to prevent double deduction
     inventoryDeductedAt: new Date()
   }
   ```

2. **InventoryTransaction** (Laboratory category):
   ```javascript
   {
     transactionType: "medical-use",
     item: glucoseStripId,
     quantity: -1,               // ← Deducted 1 strip
     reason: "Lab test completed: Glucose, Fasting",
     documentReference: labOrderId,  // ← Links to specific order
     previousQuantity: 43,
     newQuantity: 42
   }
   ```

3. **InventoryTransaction** (Service category - sync):
   ```javascript
   {
     transactionType: "medical-use",
     item: glucoseStripServiceId,
     quantity: -1,               // ← Synced with Laboratory
     reason: "Lab test completed (synced from Laboratory): Glucose Test Strips",
     documentReference: labOrderId,
     previousQuantity: 35,
     newQuantity: 34
   }
   ```

4. **Notification** (to Doctor):
   ```javascript
   {
     type: "LAB_RESULT_COMPLETED",
     message: "Lab result for John Doe is now available",
     recipient: doctorId,
     relatedDocument: {
       type: "LabOrder",
       id: labOrderId
     }
   }
   ```

Doctor sees notification → Views results → Makes diagnosis

- Sends notification to Reception

| **Recurring Tasks** | Single order | Multiple doses (one per administration) |
| **Status Flow** | Pending Payment → Ordered → Results Available | Active → Completed (per dose) |
| **Inventory Categories** | Laboratory + Service (dual) | Medication |

## 🔐 INVENTORY DEDUCTION PROTECTION

Your system has **triple-layer protection** against double deductions:

### Layer 1: Document Flag
```javascript
// Lab Orders
labOrder.inventoryDeducted = true;  // ← Prevents reprocessing

// Nurse Tasks
nurseTask.inventoryDeducted = true; // ← Prevents readministration
```

### Layer 2: Document Reference Check
```javascript
const existing = await InventoryTransaction.findOne({
  documentReference: labOrderId,  // ← Specific to this order
  item: itemId
});

if (existing) {
  return null; // Already deducted
}
```

### Layer 3: Time-Based Deduplication
```javascript
const existing = await InventoryTransaction.findOne({
  item: itemId,
  quantity: -1,
  createdAt: { $gte: new Date(Date.now() - 10000) } // Last 10 seconds
});

if (existing) {
  return null; // Recent duplicate
}
```

- Nothing happens without payment

- Doctor orders → Reception notified

- Service completed → Doctor notified

### 5. **Audit Trail**
- Every inventory deduction tracked
- Every administration recorded
- Every payment documented
- Complete chain of custody

## 🎯 SUMMARY

Your clinic workflow follows a strict **Order → Pay → Service → Inventory** pattern:

1. **Doctor creates order** (lab test or prescription)
2. **Reception processes payment** (updates status to "ready")
3. **Service provider fulfills** (lab tech or nurse)
